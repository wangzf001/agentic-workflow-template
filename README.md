# Agentic Workflow Template

> **给 AI 看的团队搭建说明书。** 当你需要为任何领域（投研、内容、开发、运营）搭建 Skill 驱动的 Agentic 团队系统时，直接用这个框架。

---

## 一、核心哲学

### 你不是工具，你是团队

```
❌ 旧范式：AI 是一个工具，你告诉它做什么
✅ 新范式：AI 是一个团队，你告诉它管什么
```

把每个 Skill 当作一个人，有独立的职责、方法论、输入输出。管理 AI 就像管理团队：分工、授权、质检。

### 三层架构（通用模式）

```
用户（CEO）
  │
  └─ 总监（调度者）
        │
   ┌────┼────┐
   ↓    ↓    ↓
 采集层  分析层  决策层
 (并行)  (并行)  (串行)
```

| 层级 | 职责 | 并行/串行 | 角色数量建议 |
|------|------|:--:|:--:|
| **采集层** | 从不同源头拉原始数据 | 并行（互不依赖） | 2-3 |
| **分析层** | 用不同方法论独立出报告 | 并行（互不污染） | 2-3 |
| **决策层** | 综合研判 → 风控 → 质检 | 串行（逐级把关） | 2-3 |

> **核心原则**：采集和分析必须并行——互不等待，互不污染。决策必须串行——逐级把关，你不会拿到没经过检查的结果。

## 二、角色设计模板

每个角色 = 一个独立人格 + 一个专属 Skill

### 角色定义模板

```yaml
角色名: 行情采集员
隶属: 采集层
职责: 一句话说清做什么
数据源/工具: 具体用什么
产出: 输出什么格式
独立性: 是否依赖其他角色（并行 or 串行）
```

### 不能犯的设计错误

| ❌ 错误 | ✅ 正确 | 原理 |
|------|------|------|
| 一个角色做太多事 | 拆成 2-3 个 | 单一职责，降低复杂度 |
| 分析角色混用方法论 | 量化/基本面/产业链独立 | 互不污染，各自验证 |
| 没有质检角色 | 必须有质量官 | 输出不倒灌 |
| 所有人串行 | 采集分析并行 | 互不等待 |
| 缺少 HITL | 关键节点暂停等确认 | 决策权永远在用户 |

## 三、工作流设计模板

### 标准工作流

```
用户指令
  ↓
总监拆解意图
  ↓
┌───┼───┐
↓   ↓   ↓       采集层并行
A   B   C
↓   ↓   ↓
└───┼───┘
  ↓
核查员（交叉验证）
  ↓
┌───┼───┐       分析层并行
↓   ↓   ↓
X   Y   Z
↓   ↓   ↓
└───┼───┘
  ↓
策略师（综合研判）  ← HITL 节点
  ↓
风控官（风险检查）
  ↓
质量官（合规+输出）
  ↓
交付
```

### 关键机制

**① 并行执行**
> 同一层内，角色之间数据互不依赖 → 同时开工

**② Reflexion Loop**
> 任一角色输出不合格 → 打回重做 → 最多 3 轮 → 仍不合格标注"尽力了"

**③ Human-in-the-Loop (HITL)**
> 关键节点暂停等用户确认：数据异常、建议大幅调仓、风控预警触发

**④ Agent 共享记忆**
> 采集的数据自动缓存，未过期直接复用，跳过采集层

## 四、Skill 编写模板

每个 Skill 对应一个角色，按以下结构编写：

```markdown
---
name: skill-name
description: 一句话说清这个技能做什么
platforms: [linux, macos]
---

# 技能名 · 角色定位

## 触发条件
- 用户说"XXX"时自动激活
- Cron 定时触发

## 执行步骤
### Step 1: ...
### Step 2: ...

## 条件分支
| 情况 | 动作 | 重试次数 |

## 合规/质检
## 存储路径
## 跨技能引用
```

## 五、设计检查清单

搭建任何团队系统时，逐条核对：

| # | 检查项 | 通过标准 |
|:--:|------|------|
| 1 | 采集层角色 ≤3 个？ | 每人只负责一种数据源 |
| 2 | 分析层角色每个用独立方法论？ | 互不污染 |
| 3 | 有核查/验证角色？ | 采集→分析中间有卡口 |
| 4 | 有质检角色？ | 输出前最后一道关 |
| 5 | 并行节点最大限度并行？ | 不互相等的都并行 |
| 6 | HITL 节点定义清晰？ | 触发条件和请示格式明确 |
| 7 | Reflexion Loop 有上限？ | 最多 3 轮 |
| 8 | 每个角色有独立 Skill？ | 一个角色一个 Skill 文件 |
| 9 | 数据缓存机制？ | 未过期数据直接复用 |
| 10 | 执行日志？ | 每步有时间+耗时记录 |

## 六、实战案例

### 案例 1：基金经理（3层10人）

```
采集层（3人并行）
  ├─ 行情采集员 → 实时行情+指数
  ├─ 财务采集员 → 财报+基本面
  └─ 资金采集员 → 北向+主力+龙虎榜

核查员（串行卡口）

分析层（3人并行）
  ├─ 量化分析师 → 七因子+超跌+择时
  ├─ 基本面分析师 → 12维深度框架
  └─ 产业链分析师 → 五维卡口+板块地图

决策层（串行）
  ├─ 策略师 → 综合研判+调仓
  ├─ 风控官 → 回撤/波动/VaR
  └─ 质量官 → 合规复核
```

### 案例 2：自媒体团队（流水线 7人）

```
内容总监 → 调度

并行层：
  ├─ 热点猎手（多平台扫描）
  └─ 选题会（评分筛选）

串行流水线：
  文案师 → 风格师+审核员（并行）→ 质检官（Reflexion Loop）→ 数据官
```

## 七、部署清单

系统搭建完成后，逐项确认：

- [ ] 所有 Skill 文件已创建并测试
- [ ] Cron 定时任务已配置（每日/每周/每月）
- [ ] HITL 决策节点已定义触发条件
- [ ] 输出路径和日志路径已配置
- [ ] 知识库摄入规则已设定
- [ ] 缓存过期时间已设定（行情15min / 财务24h）
- [ ] GitHub 仓库已推送（可复用版本）
